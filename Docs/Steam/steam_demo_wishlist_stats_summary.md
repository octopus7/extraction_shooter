# Steam 데모·위시리스트·Stats 구현 정리

## 1. Steam 출시 0순위

Steam 출시에서 0순위는 선택 기능이 아니라 **출시 가능 상태**를 만드는 것이다.

필수에 가까운 항목은 다음과 같다.

- 스토어 페이지 준비
- 빌드 업로드
- 캡슐 이미지, 스크린샷, 트레일러 등 그래픽 에셋 준비
- 가격, 패키지, 지역 설정
- 연령/심의 관련 설정
- 실행 가능한 거의 최종 빌드
- Valve 리뷰 통과

즉, “도전과제, 코옵, 리더보드” 같은 기능은 출시 필수는 아니며, 출시 가능 상태를 만든 뒤 우선순위를 따져 넣는 선택 기능이다.

---

## 2. 선택 기능 구현 우선순위

일반적인 인디 싱글/소규모 게임 기준 우선순위는 다음과 같이 보는 것이 좋다.

| 순위 | 항목 | 이유 |
|---:|---|---|
| 0 | Steam 출시 필수 체크리스트 | 안 하면 출시 불가 |
| 1 | 저장, 옵션, 안정성, 컨트롤러, Steam Cloud | 유저 리뷰 방어에 직접 영향 |
| 2 | 도전과제 | 구현 대비 효과가 좋음 |
| 3 | 데모 | 위시리스트, 피드백, 시장성 검증에 유리 |
| 4 | 리더보드/통계 | 장르가 맞으면 효과 있음 |
| 5 | 코옵/멀티 소켓 | 핵심 재미가 아니면 비용과 리스크가 큼 |
| 6 | 트레이딩 카드, 커뮤니티 장식 | 출시 후 고려 가능 |

### 핵심 결론

- **도전과제는 출시 전에 넣는 쪽 추천**
- **코옵 소켓은 핵심 재미가 아니면 출시 후로 미루는 쪽 추천**
- 출시 전 목표는 “기능 많음”이 아니라 **약속한 기능이 안정적으로 되는 것**이다.

---

## 3. 도전과제 우선순위

도전과제는 Steam 유저가 기대하는 기본적인 수집/완주 요소라서 가성비가 좋다.

추천 구성은 다음과 같다.

| 우선도 | 업적 유형 |
|---|---|
| 높음 | 스토리/챕터 클리어 |
| 높음 | 엔딩 도달 |
| 중간 | 수집, 탐험, 도전 플레이 |
| 낮음 | 반복 노가다성 카운트 업적 |
| 낮음 | 복잡한 숨겨진 조건 업적 |

출시 전에는 **10~30개 정도**면 충분하다. 너무 많으면 QA 부담이 늘고, 조건 버그가 생기면 오히려 불만 요소가 된다.

---

## 4. 코옵/멀티 소켓 우선순위

코옵은 선택 기능 중 리스크가 큰 편이다.

코옵에는 단순 소켓 구현 외에도 다음이 따라온다.

- 방 만들기, 초대, 입장 UX
- 네트워크 동기화
- 지연, 끊김 처리
- 호스트 이탈 처리
- 저장/진행도 처리
- 싱글과 다른 밸런스
- 멀티 환경 QA

### 코옵을 출시 전 구현해야 하는 경우

다음 질문에 YES가 많으면 코옵 우선순위가 올라간다.

| 질문 | YES면 |
|---|---|
| 이 게임은 혼자 하면 재미가 반감되는가? | 코옵 우선순위 높음 |
| 스토어에서 코옵을 핵심 약속으로 팔 것인가? | 출시 전 필수급 |
| 친구와 같이 하는 것이 핵심 루프인가? | 코옵 우선순위 높음 |

반대로 혼자 해도 충분히 완성도가 있다면, 코옵은 출시 후 업데이트로 미루는 것이 안전하다.

## 5. 데모 출시 수수료

일반적인 Steam 데모는 **별도 Steam Direct 수수료를 또 내지 않는다.**

| 구분 | 수수료 |
|---|---:|
| 본편 게임 App ID 생성 | Steam Direct 수수료 필요 |
| 본편에 연결된 Demo App ID | 보통 추가 수수료 없음 |
| 데모가 아니라 독립 무료 게임/프롤로그로 출시 | 별도 앱으로 취급되어 수수료 필요 가능 |

### 추천 루트

```text
본편 App ID 만들기
→ Steamworks에서 Add Demo
→ 본편에 연결된 데모 App ID 생성
→ 데모 빌드 업로드
→ 데모 출시
```

주의할 점은 “Demo”로 만들면 본편에 연결된 체험판이지만, “Prologue”, “Free Version”, “무료판”을 독립 앱처럼 만들면 별도 앱으로 취급될 수 있다는 점이다.

---

## 6. 데모 업데이트 가능 여부

데모도 출시 후 계속 업데이트할 수 있다.

일반적인 흐름은 다음과 같다.

```text
새 데모 빌드 업로드
→ 테스트용 beta branch에서 확인
→ 문제 없으면 default branch로 변경
→ 유저들이 새 데모 버전을 받음
```

### 데모 운영 방식

| 시점 | 데모 운영 |
|---|---|
| 첫 공개 | 짧고 안정적인 버전 |
| Next Fest 전 | 버그 수정, 튜토리얼 개선 |
| 본편 출시 전 | 본편으로 이어지는 세이브/홍보 동선 개선 |
| 본편 출시 후 | 데모 유지 또는 비활성화 |

데모는 한 번 내고 끝나는 것이 아니라, 본편 출시 전까지 계속 다듬는 **마케팅용 빌드**로 운영할 수 있다.

## 7. 데모와 위시리스트의 관계

데모를 내면 “이 게임이 뜰 만한지”를 어느 정도 감지할 수 있다.

정확히는 흥행 보장이 아니라, **시장 반응이 약한지 강한지 보는 출시 전 지표**다.

### 기본 판단

```text
위시리스트 반응이 좋다
→ 정식 출시가 잘될 가능성이 올라감

위시리스트 반응이 약하다
→ 정식 출시 때 갑자기 잘될 가능성은 낮음
```

물론 예외는 있다.

- 유튜버/스트리머가 갑자기 물어줌
- 출시 당일 밈이 됨
- 장르 트렌드가 갑자기 터짐
- 외부 마케팅이 매우 강하게 들어감

하지만 이런 경우는 계획 가능한 전략이라기보다는 외부 변수에 가깝다.

### 데모에서 볼 수 있는 신호

| 지표 | 의미 |
|---|---|
| 데모 페이지 방문수 | 노출/클릭이 되는가 |
| 데모 다운로드 수 | 관심을 행동으로 옮기는가 |
| 실제 플레이 수 | 받아놓고 실행하는가 |
| 플레이 후 위시리스트 전환 | 게임이 설득력 있는가 |
| 방송/영상화 여부 | 구경하는 재미가 있는가 |
| 리뷰/커뮤니티 반응 | 무엇이 막히는가 |

가장 중요한 것은 단순 다운로드가 아니라 **플레이 후 위시리스트 전환**이다.

## 8. “위시는 약한데 출시 대박”이 어려운 이유

Steam 출시 때 모든 게임이 갑자기 공평하게 크게 노출되는 구조는 아니다.

출시 전부터 다음 신호가 어느 정도 쌓여 있어야 출시 직후 반응이 붙기 쉽다.

- 위시리스트
- 페이지 방문
- 클릭률
- 데모 반응
- 외부 유입
- 팔로워
- 방송/커뮤니티 반응

그래서 데모를 냈는데도 클릭, 다운로드, 플레이, 위시리스트가 모두 약하면 정식 출시 때 갑자기 반전되기 어렵다.

### 데모 공개 후 해석

| 결과 | 해석 |
|---|---|
| 노출은 많은데 위시 전환 낮음 | 스토어 페이지, 썸네일, 장르 포지셔닝 문제 가능 |
| 다운로드는 많은데 플레이 후 반응 낮음 | 데모 초반부, 튜토리얼, 재미 전달 문제 가능 |
| 플레이는 적은데 위시가 붙음 | 컨셉은 좋은데 실행 유도 약함 |
| 방송/클립이 잘 나옴 | 성장 가능성 있음 |
| 피드백은 좋은데 숫자가 낮음 | 외부 마케팅 부족 가능성 |
| 피드백도 숫자도 낮음 | 핵심 컨셉 재검토 필요 |

데모의 진짜 목적은 무료로 나눠주는 것이 아니라 **출시 전에 망할 신호를 빨리 보는 것**에 가깝다.

## 9. Steam에서 제공되는 플레이 이탈률 지표

Steam이 기본으로 “몇 분 지점에서 몇 % 이탈” 같은 상세 게임 내부 퍼널/리텐션 차트를 제공한다고 기대하면 안 된다.

Steamworks에서 기본적으로 보기 쉬운 것은 다음과 같다.

| 지표 | Steamworks에서 보기 쉬움 | 의미 |
|---|---:|---|
| 상점 페이지 방문 | 가능 | 몇 명이 페이지를 봤는지 |
| 위시리스트 추가/삭제 | 가능 | 관심 전환 |
| 데모 다운로드 | 가능 | 체험 의향 |
| 판매/환불/매출 | 가능 | 정식 성과 |
| UTM 유입 | 가능 | 외부 마케팅 링크 성과 |
| 업적 달성률 | 가능 | 특정 지점 도달률 대용 |
| 게임 내부 몇 분 이탈 | 기본 제공 약함 | 직접 심어야 함 |

즉, Steam의 기본 지표는 주로 **상점/마케팅 퍼널** 중심이고, 게임 내부 이탈률은 직접 설계해야 한다.

## 10. 데모 이탈률 측정 방법

### 방법 1: 업적으로 구간 체크

가장 간단한 방식이다.

예시:

```text
Demo_Start
Tutorial_Clear
First_Combat_Clear
Boss_Seen
Demo_Clear
Wishlist_Click_Seen
```

이런 구간별 업적을 심으면 글로벌 업적 달성률로 대략적인 이탈 지점을 볼 수 있다.

예:

| 구간 | 달성률 |
|---|---:|
| 데모 실행 | 100% |
| 튜토리얼 완료 | 72% |
| 첫 전투 완료 | 55% |
| 보스 조우 | 31% |
| 데모 클리어 | 18% |

이렇게 보면 어느 지점에서 많이 빠지는지 감지할 수 있다.

단점은 업적은 유저에게 보이는 요소라서, 완전한 내부 로그로 쓰기에는 적합하지 않다는 점이다.

---

### 방법 2: Steam Stats로 내부 카운터 저장

Stats는 업적보다 내부 수치 저장에 가깝다.

예시:

```text
max_demo_stage_reached = 4
tutorial_completed = 1
first_combat_completed = 1
demo_completed = 0
total_demo_play_seconds = 720
death_count = 3
```

Stats는 “유저별 영구 통계값 저장 기능”에 가깝고, 리텐션 분석에도 활용할 수 있다.

다만 Mixpanel, GA4 같은 세밀한 퍼널 분석 대시보드는 아니므로, 어떤 값을 저장할지 직접 설계해야 한다.

---

### 방법 3: 외부 애널리틱스 사용

정교한 분석을 원하면 외부 분석 도구가 필요하다.

예:

- GameAnalytics
- Unity Analytics
- Firebase/GA4
- PlayFab
- 자체 서버 로그

이런 도구를 쓰면 다음을 더 잘 볼 수 있다.

- 3분 안에 나간 유저 비율
- 튜토리얼 특정 방에서의 이탈
- 첫 전투 패배 후 재시도율
- 위시리스트 유도 버튼 노출 후 클릭률
- 세션별/코호트별 리텐션

## 11. 숨김 업적의 의미

Steam의 숨김 업적은 **유저에게 영원히 안 보이는 내부 이벤트**가 아니다.

정확히는 다음에 가깝다.

```text
달성 전까지 유저에게 숨겨지는 업적
달성 후에는 유저에게 보이는 업적
```

따라서 숨김 업적은 스포일러 방지용으로는 좋지만, 완전한 리텐션 이벤트로 쓰기에는 부적합하다.

### 숨김 업적 적합도

| 용도 | 숨김 업적 적합도 |
|---|---:|
| 스토리 스포일러 방지 | 좋음 |
| 큰 진행 체크 | 가능 |
| 데모 구간별 대략 이탈률 추정 | 가능 |
| 유저에게 영원히 안 보이는 내부 로그 | 부적합 |
| 세밀한 리텐션/퍼널 분석 | 부적합 |

분석용 업적을 만든다면 이름도 유저 친화적으로 만들어야 한다.

나쁜 예:

```text
RETENTION_EVENT_03
FUNNEL_STAGE_B
PLAYER_REACHED_ROOM_07
```

좋은 예:

```text
First Step
Ready for Battle
A Taste of Adventure
Demo Complete
```

## 12. Steam Stats의 성격

Steam Stats는 리텐션 분석 전용 도구라기보다는 **유저별 영구 통계값 저장 기능**이다.

원래 용도는 다음에 가깝다.

| 용도 | 예시 |
|---|---|
| 누적 플레이 기록 | 총 처치 수, 최고 점수, 클리어 횟수 |
| 업적 조건 계산 | 적 100마리 처치 시 업적 해금 |
| 리더보드 보조 | 최고 점수, 최고 기록 |
| 유저별 진행 통계 | 최고 도달 스테이지 |
| 글로벌 통계 | 전체 유저 평균/누적 값 |

이 기능을 데모 리텐션 측정에도 활용할 수 있다.

### Stats와 리텐션 분석의 차이

| 기능 | Steam Stats |
|---|---:|
| 유저별 수치 저장 | 가능 |
| 업적 조건과 연동 | 가능 |
| 글로벌 통계 요청 | 가능 |
| 세밀한 퍼널 차트 | 약함 |
| 실시간 이벤트 로그 | 부적합 |
| 종료 지점 기록 | 직접 설계 필요 |
| 세션별 리텐션 분석 | 부적합 |
| 코호트 분석 | 부적합 |

정리하면:

```text
Steam Stats = 내부 통계값 저장소
리텐션 분석 = 그 저장소를 이용해 직접 설계해야 하는 분석
```

## 13. 데모 분석용 최소 Stats 세트

초기 인디 기준으로는 아래 정도면 충분하다.

```text
max_demo_stage_reached
tutorial_completed
core_fun_reached
demo_completed
total_demo_play_seconds
wishlist_prompt_seen
wishlist_prompt_clicked
death_count
```

### 추천 의미

| Stat | 의미 |
|---|---|
| `max_demo_stage_reached` | 유저가 데모에서 최대 어디까지 갔는가 |
| `tutorial_completed` | 튜토리얼을 넘겼는가 |
| `core_fun_reached` | 게임의 핵심 재미 지점까지 도달했는가 |
| `demo_completed` | 데모를 끝까지 완료했는가 |
| `total_demo_play_seconds` | 누적 플레이 시간 |
| `wishlist_prompt_seen` | 위시리스트 유도 화면을 봤는가 |
| `wishlist_prompt_clicked` | 위시리스트 버튼을 눌렀는가 |
| `death_count` | 데모 중 사망 횟수 |

중요한 것은 데모 클리어율보다 **핵심 재미 도달률**이다.

데모를 끝까지 못 해도 5~10분 안에 “이거 괜찮은데?”가 오면 위시리스트로 이어질 수 있다.

---

## 14. Steam Stat의 증가 전용 제약

Steam Stat에는 **Increment Only** 옵션을 걸 수 있다.

이 옵션을 켜면 해당 Stat은 시간이 지나며 **증가만 가능**하다.

### Increment Only에 적합한 값

| Stat | Increment Only 추천 |
|---|---:|
| `total_kills` | 좋음 |
| `total_play_seconds` | 좋음 |
| `demo_clear_count` | 좋음 |
| `max_demo_stage_reached` | 좋음 |
| `highest_score` | 좋음 |
| `death_count` | 좋음 |
| `tutorial_completed` | 좋음 |
| `demo_completed` | 좋음 |

### Increment Only에 부적합한 값

| Stat | Increment Only 비추천 |
|---|---:|
| `current_hp` | 비추천 |
| `current_stage` | 비추천 |
| `current_money` | 비추천 |
| `last_selected_character` | 비추천 |
| `session_play_seconds` | 비추천 |

데모 리텐션용으로는 `max_stage`, `completed_flag`, `total_count` 류에 특히 잘 맞는다.

---

## 15. Steam Stat의 다른 제약 조건

Steam Stat에는 다음 같은 제약을 설정할 수 있다.

| 설정 | 의미 |
|---|---|
| Min Value | 최소값 제한 |
| Max Value | 최대값 제한 |
| Max Change | 한 번의 변경에서 허용되는 최대 변화량 |
| Default Value | 신규 유저 기본값 |
| Set By | Client / Game Server / Official Game Server 등 누가 수정 가능한지 제한 |
| Increment Only | 값이 증가만 가능하게 제한 |

데모 리텐션용 Stat에는 이런 식으로 방어를 걸 수 있다.

```text
max_demo_stage_reached: Increment Only, Min 0, Max 데모최대스테이지
tutorial_completed: Increment Only, Min 0, Max 1
core_fun_reached: Increment Only, Min 0, Max 1
demo_completed: Increment Only, Min 0, Max 1
total_demo_play_seconds: Increment Only, Min 0, Max 적당히 큰 값
death_count: Increment Only, Min 0
```

---

## 16. Stat 값 변경은 클라이언트 코드 책임

Stat 값의 “언제, 얼마로 바꿀지”는 기본적으로 클라이언트 코드 책임이다.

Steam은 다음을 제공한다.

- 저장소
- 읽기/쓰기 API
- 제약 조건
- 업적/통계 연동

하지만 어떤 순간에 어떤 값을 `SetStat`할지는 게임 코드가 결정한다.

기본 흐름은 다음과 같다.

```text
RequestCurrentStats
→ GetStat
→ 로컬에서 비교/계산
→ SetStat
→ StoreStats
```

---

## 17. 메모리에서 값이 날아가면 어떻게 되는가

상황별로 다르다.

| 상황 | 결과 |
|---|---|
| Steam 서버에 이미 `StoreStats()` 완료됨 | 다음 실행 때 다시 불러올 수 있음 |
| 메모리에서만 바꾸고 `StoreStats()` 안 함 | 게임 종료/크래시 시 날아갈 수 있음 |
| `RequestCurrentStats()` 전에 기본값 0으로 덮어씀 | 기존 Steam 값을 낮은 값으로 저장할 위험 있음 |
| `Increment Only` 켜져 있음 | 낮은 값으로 덮어쓰는 것을 방어 가능 |
| Min/Max/Max Change 설정 있음 | 이상값 일부 방어 가능 |

중요한 점은 다음이다.

```text
메모리에서 값이 날아간다고 Steam 서버 값이 자동 리셋되는 것은 아니다.
하지만 클라이언트가 0 같은 잘못된 값을 SetStat + StoreStats 하면 리셋처럼 보일 수 있다.
```

---

## 18. 가장 위험한 실수

기존 Steam 서버 값이 다음과 같다고 하자.

```text
max_demo_stage_reached = 5
```

그런데 게임 시작 시 Steam 값을 읽기 전에 로컬 기본값을 0으로 둔다.

```text
max_demo_stage_reached = 0
```

그리고 종료 시 무조건 `StoreStats()`를 해버리면, 제약이 없다면 5를 0으로 덮어쓸 수 있다.

그래서 중요한 값은 다음처럼 처리해야 한다.

```text
항상 Steam에서 현재 값을 먼저 읽는다.
새 값과 기존 값을 비교한다.
더 큰 값만 저장한다.
가능하면 Increment Only도 설정한다.
```

## 19. 안전한 구현 패턴

### 최대 도달 스테이지 예시

```csharp
// 개념 예시
SteamUserStats.RequestCurrentStats();

// stats 로드 완료 콜백 이후
int current;
SteamUserStats.GetStat("max_demo_stage_reached", out current);

int newValue = Mathf.Max(current, reachedStage);

if (newValue > current)
{
    SteamUserStats.SetStat("max_demo_stage_reached", newValue);
    SteamUserStats.StoreStats();
}
```

### 플래그형 Stat 예시

```text
tutorial_completed = max(current, 1)
core_fun_reached = max(current, 1)
demo_completed = max(current, 1)
wishlist_prompt_seen = max(current, 1)
```

### 저장 타이밍

Steam Stats는 매 프레임이나 매 초 보내는 로그용이 아니다.

추천 저장 시점은 다음과 같다.

| 시점 | 저장 여부 |
|---|---|
| 튜토리얼 완료 | 저장 |
| 첫 전투 완료 | 저장 |
| 스테이지 진입 | 저장 |
| 데모 클리어 | 저장 |
| 게임 종료 직전 | 저장 |

`total_demo_play_seconds` 같은 값은 로컬에서 누적하다가 체크포인트, 종료 시점, 스테이지 전환 시점에 저장하는 식이 좋다.

## 20. 최종 추천 구조

### 출시 전 선택 기능 우선순위

```text
안정성/세이브/옵션
→ Steam Cloud
→ 도전과제
→ 데모
→ 리더보드/Stats
→ 코옵
```

단, 게임이 친구와 함께 해야 핵심 재미가 살아나는 구조라면 다음처럼 바뀐다.

```text
코옵 핵심 루프
→ 안정성
→ 초대/방 UX
→ 저장/진행도
→ 도전과제
```

### 데모 분석 최소 세트

```text
Steam Achievements:
- Demo Complete
- First Victory
- Secret Encounter

Steam Stats:
- max_demo_stage_reached
- tutorial_completed
- core_fun_reached
- demo_completed
- total_demo_play_seconds
- wishlist_prompt_seen
- wishlist_prompt_clicked
- death_count
```

### 결론

- 도전과제는 출시 전 구현 추천
- 코옵은 핵심 재미가 아니면 출시 후로 미루는 것이 안전
- 데모는 추가 수수료 없이 본편에 연결해 낼 수 있음
- 데모는 출시 후에도 계속 업데이트 가능
- 데모 반응과 위시리스트 증가율은 출시 전 시장성 판단에 중요
- Steam은 세밀한 내부 이탈률을 기본 제공하지 않음
- 이탈률은 업적, Stats, 외부 애널리틱스로 직접 설계해야 함
- 숨김 업적은 완전한 내부 로그가 아니라 달성 전까지만 숨겨지는 업적
- Steam Stats는 내부 통계 저장소이며, 리텐션 분석에도 활용 가능
- Stat 값 변경 로직은 클라이언트 코드 책임
- 안전하게 쓰려면 `RequestCurrentStats` 이후 `max(old, new)` 패턴과 `Increment Only`를 같이 쓰는 것이 좋음
